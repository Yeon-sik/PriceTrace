## 1. 문제 정의와 제품 방향

### 해결하려는 문제

| 사용자 문제 | PriceTrace의 접근 |
| --- | --- |
| 장을 본 뒤 실제 구매 가격을 다시 비교하기 어렵다. | 영수증 항목을 가격 관측값으로 기록하고, 상품·판매처·관측일별 이력을 조회한다. |
| 판매처마다 같은 상품 이름과 표기가 달라 단순 이름 검색은 신뢰하기 어렵다. | 판매처 상품 코드와 표준 상품 카탈로그를 분리하고, 검토된 매핑만 통합 비교에 사용한다. |
| 장보기 전 예상 비용을 가늠하기 어렵다. | 관측가를 기준으로 상품을 장바구니에 담아 예상 합계를 계산한다. |
| 공동 구매 물품의 수량 배분과 입금 확인이 번거롭다. | 수령자별 수량 배분, 전달·입금 상태, 정산 메시지 생성을 지원한다. |
| 영수증 데이터는 개인 구매 이력이므로 공유 경계가 중요하다. | 원본·가변 상태·공개 샘플을 분리하고, 원격 데이터에는 사용자별 접근 제어를 둔다. |

### 제품 원칙

1. 가격은 실시간 시세가 아니라 **영수증 관측가**다.
2. 상품명은 식별자가 아니다. 판매처 상품 코드와 검토된 표준 상품 매핑을 사용한다.
3. 검증되지 않은 검색 결과·상품 매핑·가격은 확정 데이터처럼 표시하지 않는다.
4. 원본 영수증은 읽기 전용 구매 사실로 보존하며, 정산 상태는 별도로 관리한다.
5. 금액은 KRW 정수로 계산한다.

## 2. 대상 사용자와 핵심 사용자 흐름

### 대상 사용자

| 우선순위 | 사용자 | 필요한 가치 |
| --- | --- | --- |
| 1차 | 여러 마트 또는 PX에서 구매하는 개인·가구 | 실제 구매가 기반의 가격 기록, 비교, 장보기 예상 금액 |
| 1차 | 공동 구매를 하는 친구·소규모 모임 | 물품 수량 배분, 사람별 정산 금액, 입금·전달 상태 관리 |
| 2차 | 표준 상품과 판매처 상품을 연결하는 운영자 | 상품 매핑의 검토·승인·반려와 출처 관리 |
| 2차 | 가격 데이터 품질을 점검하는 관리자 | 품질 플래그, 감사 로그, 검토 가능한 데이터 경계 |

### 대표 사용자 흐름

```text
영수증 데이터 선택
  → 상품 검색 또는 카테고리·판매처 필터
  → 상품별 관측가와 가격 이력 확인
  → 필요한 상품을 장바구니에 추가
  → 예상 구매 금액 확인
  → 공동 구매인 경우 수령자·수량을 배분하고 정산 상태 관리
```

> 1. 상품 검색·필터 화면  2. 가격 이력/비교 화면  3. 공동 정산 또는 장바구니 화면

| 영역 | 선택 | 선택 이유 | 트레이드오프 |
| --- | --- | --- | --- |

| 초기 저장 | localStorage | 로그인·서버 설정 없이도 정산 MVP를 바로 검증한다. 원본 영수증과 사용자의 가변 상태를 분리하는 로컬 우선 전략이다. | 기기 간 동기화·복구에 한계가 있어 백업·복원과 원격 동기화 단계가 필요하다. |

## 5. 핵심 기술 의사결정

### 결정 1. 로컬 우선 MVP 후 원격 동기화로 확장

**선택**: 초기 정산 기능은 JSON + localStorage로 시작하고, 가격 이력·다중 기기·사용자 격리가 필요한 영역을 Supabase로 확장한다.

**이유**: 정산의 핵심 가치는 로그인이나 서버 없이도 검증할 수 있다. 초기 단계에서 인프라를 먼저 만들면 사용자 흐름 검증이 늦어진다.

**결과와 후속 과제**: 로컬 데이터 손실 위험을 백업/복원으로 완화하고, 원격 동기화가 추가된 이후에는 충돌 정책·오프라인 정책·동기화 실패 UX를 명확히 해야 한다.

### 결정 2. 상품명 대신 판매처 상품 코드와 표준 상품 매핑 사용

**선택**: 상품명만으로 동일 상품을 판정하지 않는다.

**이유**: 같은 이름이라도 규격·용량·묶음 수·판매처 코드가 다르면 가격 비교가 왜곡된다.

**결과와 후속 과제**: 데이터 신뢰도는 높아지지만 매핑 검토 비용이 생긴다. 자동 매핑을 도입하더라도 `pending → verified/rejected`의 사람 검토 경로를 유지한다.

### 결정 3. 검색 결과는 후보이며 자동 확정하지 않음

**선택**: Brave Search API 결과를 공식 상품 후보로만 제공한다.

**이유**: 검색 엔진 결과만으로 상품 규격·브랜드·구성을 확정하면 잘못된 가격 비교가 발생할 수 있다.

**결과와 후속 과제**: 자동화율은 낮아지지만 데이터 신뢰도를 우선한다. 사용자가 확인할 URL·브랜드·규격을 명확히 보여 주어 검토 비용을 낮춘다.

### 결정 4. 정적 웹 배포와 서버리스 비밀 관리 분리

**선택**: UI는 GitHub Pages 정적 배포를 사용하고, 비밀 키가 필요한 상품 검색은 Supabase Edge Function에서 실행한다.

**이유**: 정적 사이트의 간단한 배포 이점을 유지하면서 API 키를 클라이언트에 노출하지 않는다.

**결과와 후속 과제**: 원격 기능의 장애와 API 사용량은 별도로 관찰해야 하며, Edge Function의 인증·오류·요청 제한 정책을 운영 기준으로 정리해야 한다.

## 6. 아키텍처와 데이터 흐름

```text
영수증 JSON / Supabase PostgreSQL
        │
        ▼
Zod Schema Validation
        │
        ▼
Repository / Mapper
  ├─ JSON Receipt Repository
  ├─ LocalStorage Settlement Repository
  └─ Supabase Settlement Repository
        │
        ▼
Domain Service / Selector
  ├─ 영수증 금액·수량 불변식
  ├─ 배분 가능 수량 검증
  ├─ 수령자별 정산 합계
  └─ 가격 이력·판매처 비교
        │
        ▼
Next.js / React UI
```

| 가격 관측 | `store_products`, `price_observations` | 판매처 상품 코드와 영수증 기반 관측가 이력 |
| 표준 카탈로그 | `catalog_categories`, `catalog_products`, `source_product_mappings` | 서로 다른 판매처 상품을 비교하기 위한 표준 상품과 검토 상태가 있는 매핑 |
| 공동 정산 | `recipients`, `allocations`, `settlement_statuses` | 수령자, 항목별 배분 수량, 전달·입금 상태 |
| 운영 | `receipt_quality_flags`, `audit_logs` | 데이터 품질 점검과 운영 이력 |

### 도메인 불변식

- `unitPriceKrw >= 0`
- 구매 수량과 배분 수량은 양의 정수
- `totalPriceKrw = unitPriceKrw × purchasedQuantity`
- 품목별 배분 수량 합은 구매 수량을 초과할 수 없음
- 친구별 합계는 배분 내역에서 계산하며 별도 총액을 진실 원천으로 저장하지 않음
- 미배분 수량은 구매 수량에서 배분 수량 합을 뺀 값
- 원본 구매번호 배열을 보존

## 10. 포트폴리오 관점의 기여와 배운 점

### 이 프로젝트에서 보여 주는 역량

- 단순 상품 목록이 아니라 가격의 **출처·시점·검증 상태**를 모델링하는 도메인 설계
- localStorage MVP에서 인증·RLS·관계형 데이터 모델로 확장하는 단계적 아키텍처 설계
- 상품명만으로 동일 상품을 판단하지 않는 데이터 품질 기준 수립
- 외부 검색 API의 보안 경계와 사람 검토 절차를 함께 설계
- 도메인 규칙을 UI 밖으로 분리하고 단위 테스트로 검증

### 개선이 필요한 지점

- 실제 사용자 인터뷰, 활성 사용자 수, 가격 데이터 축적량은 아직 제품 성과 지표로 기록되지 않았다.
- 정산 UI와 원격 동기화의 전체 사용자 흐름은 배포 환경에서 재검증이 필요하다.
- OCR, 알림, 단위 가격 비교는 데이터 품질과 운영 기준을 확보한 뒤 확장해야 한다.
- 정적 배포와 원격 기능의 장애 경계, 백업·복구 절차를 운영 문서와 자동 점검으로 보강해야 한다.

### 다음에 다시 설계한다면

1. 초기부터 동기화 충돌 정책과 오프라인 상태 표기를 사용자 흐름에 포함한다.
2. 상품 매핑 검토 대기열과 관리자 UX를 먼저 명확히 만들어 데이터 축적 비용을 낮춘다.
3. 상품 규격·단위 데이터를 확보한 이후에만 단위 가격 비교를 도입한다.
4. 기능 확장보다 실제 영수증 기반 사용자 검증과 데이터 품질 지표를 먼저 수집한다.

## 공식 상품 연결 운영 방침(현재 적용)

공식 상품 자동 탐색 및 자동 연결은 보류한다. 외부 상품 검색 API를 호출하는 방식은 검색 결과의 정확도가 충분하지 않아, 잘못된 상품을 연결하거나 가격 비교 대상을 오인할 위험이 있다.

당분간은 영수증에서 확인된 PX(군마트)와 일반 마트 품목만 대상으로 사용자가 직접 상품을 비교하고 연결한다. 기존에 설계된 공식 상품 탐색 화면, 후보 확인 흐름, 저장 구조는 유지하되 자동 검색 결과를 공식 상품으로 확정하지 않는다.

수동 연결의 기본 흐름은 다음과 같다.

1. 영수증 품목의 판매 마트 정보와 마트별 상품 코드를 확인한다.
2. PX 품목과 일반 마트 품목을 각각 비교하고, 동일 상품으로 판단되는 항목을 사용자가 직접 선택한다.
3. 상품명만으로 확정하지 않고 브랜드, 용량·수량, 포장 단위, 상품 코드 등 식별 정보를 함께 확인한다.
4. 연결되지 않거나 대응 상품을 찾지 못한 품목은 미탐색·미연결 상태로 별도 분류한다.
5. 수동 확인이 완료된 연결만 가격 추이와 판매처 비교의 기준 데이터로 사용한다.

이 방침은 공식 상품 연결 기능의 삭제가 아니라 검증 방식의 변경이다. 검색 API 연동은 향후 검색 정확도와 검증 절차가 확보될 때 재검토한다.

## 최근 반영: 범용 영수증과 PX 공통 카탈로그

영수증 입력 형식은 판매처마다 달라지는 화면용 구조가 아니라 `receipt.v2` 범용 모델로 통일했다. 문서·판매처·품목·합계·결제 정보를 분리해 원본 증거를 보존하며, 금액은 KRW 정수로 처리한다.

판매처에는 다음 정보를 함께 기록한다.

- `merchant.retail_channel`: `px`, `regular`, `unknown`
- `merchant.catalog_namespace`: 여러 매장이 공유한다고 검증된 상품 코드 체계. 확인되지 않았으면 `null`

상품명은 식별자가 아니다. 서로 다른 매장의 상품을 자동으로 하나로 보려면 다음 조건이 모두 맞아야 한다.

1. 동일한 `catalog_namespace`
2. 동일한 판매처 상품 코드
3. 동일한 정규화 상품명

따라서 PX 공통 카탈로그에서 같은 코드와 이름이 확인된 상품은 공식 상품 연결 전에도 하나의 카드로 표시하고, 판매처와 관측 횟수를 함께 보여 준다. 반대로 카탈로그가 다르거나 용량·규격이 다른 경우에는 코드 또는 이름이 일부 같아도 자동 병합하지 않는다.

공식 상품으로 연결된 뒤에는 같은 공식 상품 URL을 기준으로 판매처 품목을 통합해 가격 이력을 표시한다. 공식 연결 전의 일반 판매처 상품은 같은 판매처 안에서만 가격 이력을 비교한다.
